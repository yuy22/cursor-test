# RAG 知识库构建 — 数据清洗完整操作指南

> **适用人群**：任何需要把书籍/文档清洗成 RAG 友好 Markdown 的 AI 或开发者
> **目标**：给一个扫描书籍 DOCX 或 PDF，输出可直接投入 RAG 系统的 Markdown（含图片描述）
> **工具包包含**：本指南 + clean_docx_v3.py 脚本 + 常见问题手册

---

## 一、判断文件类型，选对流水线

```
拿到文件
    │
    ├── .docx（OCR扫描版 Word）→ 走【流水线 A】clean_docx_v3.py
    │
    ├── .pdf（原版 PDF）       → 走【流水线 B】TextIn OCR → fix_md_headings.py
    │
    └── 已是 .md（原始 OCR）  → 直接从流水线 B 的第二步开始
```

> **如何判断 DOCX 是"扫描版"？** 打开后文字可以复制但排版混乱、有大量图片、字体不统一，即为 OCR 扫描版。

---

## 二、流水线 A：扫描版 DOCX → Markdown

### 第一步：字号分析（必做，不能跳过）

```bash
python clean_docx_v3.py --analyze
```

**输出示例：**
```
字号(pt)  加粗  次数  示例文本
23.5pt   True    4   教育是一门艺术      ← ## 级别（章标题）
16.0pt   True    2   1998—2002 忽如一夜  ← ### 级别（节标题）
13.5pt  False   33   访谈手记            ← #### 级别（子标题）
★17.0pt False    1   对于可放一          ← ★ 危险！孤立字号，人工确认
10.5pt  False 1065   责任编辑 刘 灿...   ← 正文/噪声（最多的那个）
```

**判断规则：**
- 出现次数最多的字号 → 正文，不做标题
- 次数较多、字号较大、加粗 → 章节标题（## 或 ###）
- ★ 标记（出现次数 ≤3） → 必须人工确认是否是 OCR 乱码

---

### 第二步：配置脚本顶部

打开 `clean_docx_v3.py`，只改以下区域：

```python
INPUT        = Path("你的文件.docx")          # 输入文件路径
OUT_DIR      = Path("output/书名_cleaned")     # 输出文件夹
BOOK_TITLE   = "书名"                          # 书名
BOOK_AUTHOR  = "作者名"                        # 作者

# 根据第一步的分析结果填写（字号 → Markdown 标题级别）
SIZE_TO_HEADING = {
    23.5: 2,   # → ## 大章标题
    16.0: 3,   # → ### 节标题
    13.5: 4,   # → #### 子标题
}

# 防止 OCR 字号漂移：这些文本无论什么字号都强制识别为 #### 标题
KNOWN_HEADINGS_H4 = {
    '访谈手记',    # 示例：填入本书反复出现的固定小标题
}

# 本书特有的页眉正则（书名/作者在页眉中反复出现的行）
BOOK_HEADER_PATTERNS = [
    # re.compile(r'^书名\s*$'),
]
```

**字号配置经验：**
- 如果同一类标题字号漂移（如 13.0/13.5/14.0pt 都是同一个标题），把相邻字号全加进来
- 不确定时先留空 `{}`，脚本会退回文字正则（效果差一些但能跑）

### 第三步：正式清洗

```bash
python clean_docx_v3.py
```

**自动完成：**
- 噪声过滤（水印、页码、出版信息、孤立人名）
- 文本修复（压缩中文间空格、半角标点→全角、去末尾页码）
- 结构识别（按字号映射转为 Markdown 标题）
- 表格转 Markdown
- 图片提取分类（自动删除装饰碎片、分隔线、空白图）

**质量报告（跑完必看）：**
```
## 章节: 19    ← 如果 < 5 个，配置有误，回第二步
### 小节: 33
#### 子节: 140
```

---

### 第四步：图片审查（必做）

```bash
python clean_docx_v3.py --review-images
```

生成 `image_review/审查报告.html`。

**操作流程：**
1. **先解压**输出的 zip，再双击 HTML（不能在 zip 内直接打开）
2. 逐张检查"保留的图"，分三类处理：

| 类型 | 判断标准 | 处理 |
|---|---|---|
| 垃圾图 | 书脊装饰、空白页、灰色色块 | 删除 `![图片](...)` 整行 |
| 装饰可转文字 | 签名、标题装饰 | 替换为文字 |
| 有教学价值 | 教学插图、图表、公式图 | 写描述 + 保留图片引用 |

3. 也要检查"删除的图"中边界附近的图（亮度 225~235、面积 3000~6000px²），可能有误删

---

### 第五步：写图片描述（核心步骤）

**格式：**
```markdown
[图片描述：描述内容写在这里]
![图片](images/rId109.jpeg)
```

**写描述的正确视角（重要！）：**

❌ 错误：只描述外观（"一个圆形，里面有数字"）
✅ 正确：从教学视角描述（"展示计数器，用于帮助学生理解数位概念"）

追问自己三个问题：
1. 这张图在**教什么**？
2. 用的是**什么方法/思路**？
3. 学生能从这里**学到什么**？

**特殊情况处理：**

| 图片类型 | 描述方式 |
|---|---|
| 数学公式图 | 用文字还原算式："8÷4 = 8×(1/4) = 2，展示除以整数等于乘以倒数" |
| 竖式计算 | 逐步描述各行："第一步3×113=339，第二步20×113=2260，合计2599" |
| 几何示意图 | 描述形状+教学目的："梯形被虚线分成两个三角形，演示面积分割计算" |
| 统计图表 | 描述数据范围+结论："2018-2020年优良天数从227增至276，说明空气质量改善" |
| 扫描模糊 | 诚实写："扫描模糊，请对照原书确认，大致为XXX" |
| 装饰图 | "装饰性图片，无教学内容" |

### 第六步：二次清洗（第一轮跑完后必做）

检查漏网噪声：

```bash
# 快速扫描1~6字纯汉字短行（可能是噪声或裸文本标题）
python -c "
import re
lines = open('output/书名_cleaned/书名_cleaned.md',encoding='utf-8').readlines()
for i,l in enumerate(lines,1):
    s=l.strip()
    if re.match(r'^[\u4e00-\u9fff]{1,8}$',s) and not s.startswith('#'):
        print(f'L{i}: {s}')
"
```

常见漏网情况：
- OCR 空格变体：`出 版 人 策划编辑` → 用 `fix_text()` 后再过一遍
- 目录行：含 `/数字` 格式 → 正则 `.+/\d+\s*$` 过滤
- 孤立人名：`^[\u4e00-\u9fff]{2,3}$` 匹配（注意不删 `#` 开头的标题行）

### 第七步：标题层级审查（按照原书目录校对）

**最常见的标题层级问题：**

1. **OCR 字号漂移** → 同一标题出现多种字号，导致标题漏识别
   - 解决：在 `KNOWN_HEADINGS_H4` 里固定文本

2. **章节编号与标题分离** → `#### 第一章` 和 `## 标题名` 分成两行
   - 解决：手动合并 `## 第一章 标题名`

3. **子标签被抬升** → `学习活动`/`学习目标`/`课后总评` 变成 `##` 标题
   - 解决：降级为 `####` 或正文粗体

4. **章节没有中间层** → 章（##）直接跳到最小标题（####），看不到子结构
   - 解决：如果该章无分节，问题标题应升为 `###`

**标准层级结构：**
```
# 书名
## 第一章 章名            ← 最高结构层
  ### 第一节 节名          ← 有分节时
    #### 问题X 问题标题    ← 最小内容单元
  ### 问题X 问题标题       ← 无分节时直接在章下
```

## 三、流水线 B：PDF → Markdown（TextIn + 后处理）

### 步骤概览

```
PDF
 → TextIn 云服务 OCR（需网络）
 → 原始 MD（含 textin.com 外链图片）
 → fix_md_headings.py（标题层级 + 噪声 + 面包屑前缀）
 → process_images.py --phase download（下载图片到本地）
 → process_images.py --phase describe（Vision API 生成图片描述）
 → process_images.py --phase update（外链→本地路径 + alt text）
 → process_images.py --phase report（生成质量审核报告）
 → process_images.py --phase fix_alt（修复垃圾 alt text）
 → 最终 *_RAG优化.md
```

### 图片描述格式（流水线 B）

图片描述放在 `alt text` 里，不另起行：
```markdown
![正方形被平均分成10份，其中3份涂色，表示3/10=0.3](C:\...\images\book1_042.jpg)
```

### Vision API 配置

```python
MODEL = "claude-opus-4-6"        # 推荐用 Opus 而不是 Haiku（Haiku 描述质量差）
max_tokens = 2048                 # 不能太少，否则描述被截断
```

**Prompt 三条核心规则：**
1. 先独立观察图片内容，再对照上下文判断是否相关，相关才融入
2. 图片中所有文字必须说明位置和含义，禁止留空白
3. 看不清的写"此处文字模糊，请参看原图"，禁止猜测

## 四、数学公式处理（两种流水线通用）

| 公式类型 | 处理方式 |
|---|---|
| `√ × ÷ ± → ≤ ≥ ≠ ℃` 等 Unicode | **直接保留**，所有系统原生支持 |
| OCR 乱码（`∘`代替`°`，`\sdiv`代替`÷`） | 用正则批量修复 |
| LaTeX 分数 `$\frac{1}{10}$` | 转中文语义：十分之一（1/10）|
| 图片里的公式 | 用文字描述还原算式内容 |
| OMML 公式（Word 内嵌公式对象） | 提取为图片后写描述 |

**常见 OCR 公式乱码修复正则：**
```python
import re
text = re.sub(r'(\d+)∘', r'\1°', text)      # 角度: 60∘ → 60°
text = re.sub(r'∘C', '°C', text)             # 温度: ∘C → °C
text = re.sub(r'\\sdiv', '÷', text)          # 竖式除法
text = re.sub(r'\\line\b', '', text)          # 竖式横线
text = re.sub(r'\\\\', '\\', text)           # 过度转义
```

## 五、RAG 优化原则

| 原则 | 说明 |
|---|---|
| **不去停用词** | 现代 Embedding 模型需要完整自然语言，去停用词反而降低检索质量 |
| **图片必须有文字描述** | 标准 RAG 框架不传图片，只有描述文字能被检索 |
| **保留原图路径** | 未来多模态 RAG 可直接用 |
| **公式转中文语义** | `$\frac{1}{10}$` → "十分之一（1/10）"，中文语义更易检索 |
| **面包屑前缀** | 章节标题加路径标注（如 `> 第一章 > 第一节`），帮助 RAG 定位上下文 |

## 六、质量检查清单（清洗完成后逐项确认）

- [ ] `##` 标题数量合理（根据书的体量，通常 ≥ 5 个）
- [ ] `###` / `####` 标题层级与原书目录一致
- [ ] 孤立字号（★ 标记，出现次数 ≤3）已人工确认
- [ ] 图片：保留的图人工审查；删除的图边界附近已抽查
- [ ] 图片描述：从教学视角写，不只描述外观
- [ ] 无 `学习活动`/`学习目标` 等子标签被抬升为章级标题
- [ ] 无明显语义断裂的1~6字孤立汉字行
- [ ] 数学公式 OCR 乱码已修复（`∘`/`\sdiv`/`\line`）
- [ ] 不含水印、页码、出版社信息等噪声

## 七、常见问题 Q&A

**Q：中文路径报 PackageNotFoundError？**
A：不要用 bash heredoc 传路径，改用 Write 工具把 Python 脚本写成文件再运行。

**Q：`## ` 标题 < 5 个，大概率配置有误？**
A：先运行 `--analyze` 查看字号分布，确认 `SIZE_TO_HEADING` 配置是否正确。

**Q：表格内容有重复列？**
A：DOCX 合并单元格在 python-docx 中会重复，属正常现象，不影响语义。

**Q：图片过滤后还有垃圾图？**
A：自动分类不可能 100% 准确，必须人工逐张审查。

**Q：扫描质量太差看不清图片内容？**
A：用 pymupdf 导出 PDF 对应页查看，描述必须忠实，看不清就写"扫描模糊"。

**Q：Vision API 描述质量差（误判/截断/垃圾）？**
A：换用更强的模型（Opus > Sonnet > Haiku），增大 max_tokens（建议 ≥ 2048），优化 Prompt（加"先看图再看上下文"等规则）。

**Q：某个标题漏识别了（OCR 字号漂移）？**
A：把该标题的精确文本加入 `KNOWN_HEADINGS_H4`，无论什么字号都强制识别。
