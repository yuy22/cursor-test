# 数据清洗方法手册

> 本文档记录对扫描版/OCR书籍 DOCX 进行清洗并转为 RAG 友好 Markdown 的完整方法。
> 可直接调用桌面上的脚本，无需重新开发。

### 标准五步流程（每本新书都按这个顺序）

**第一步：配置脚本顶部**（只改这里，其余不动）
```python
INPUT        = Path("你的文件.docx")
OUT_DIR      = Path("output/书名_cleaned")
BOOK_TITLE   = "书名"
BOOK_AUTHOR  = "作者名"
SIZE_TO_HEADING = {}      # 先留空，第二步填
KNOWN_HEADINGS_H4 = set() # 先留空，第三步填
BOOK_HEADER_PATTERNS = [] # 本书特有页眉正则，可选
```

**第二步：字号分析（必做！不要跳过）**
```bash
python3 clean_docx_v2.py --analyze
```
输出示例：
```
字号(pt)  加粗  次数  示例文本
23.5pt   True    4   教育是一门艺术      ← ## 级别
16.0pt   True    2   1998—2002 忽如一夜春风来  ← ### 级别
13.5pt  False   33   访谈手记            ← #### 级别
★17.0pt False    1   对于可放一           ← ★ 危险！孤立字号，人工确认是否乱码
10.5pt  False 1065   责任编辑 刘 灿...   ← 正文/噪声
```
★ 标记的是出现次数 ≤3 的孤立字号，**必须人工确认**是否是 OCR 乱码。

**第三步：根据分析结果填写 SIZE_TO_HEADING**
```python
SIZE_TO_HEADING = {
    23.5: 2,   # 教育是一门艺术
    16.0: 3,   # 1998—2002 忽如一夜春风来
    13.5: 4,   # 访谈手记
}
# 同一类标题如果有字号漂移（如访谈手记出现13/13.5/14pt），
# 把相邻字号都加进来，或者用 KNOWN_HEADINGS_H4 固定
KNOWN_HEADINGS_H4 = {'访谈手记'}  # 无论字号，这些文本固定为 ####
```

**第四步：正式清洗**
```bash
python3 clean_docx_v2.py
```
清洗报告会打印标题数量。**如果 `## ` 标题 < 5 个，大概率配置有误**，回去检查字号。

**第五步：图片审查**
```bash
python3 clean_docx_v2.py --review-images
```
生成 `image_review/审查报告.html`，解压后用浏览器打开，看哪些图被误删。
确认后告知 AI 哪些图要加回 MD。

输出结构：
```
书名_cleaned/
├── 书名_cleaned.md      ← 清洗后的 Markdown（相对路径图片）
├── images/              ← 保留的图片
│   └── rIdXX.jpeg
└── image_review/        ← 图片审查报告（--review-images 生成）
    ├── 审查报告.html    ← 解压后双击打开
    ├── 01_kept/
    ├── 02_deleted_small/
    ├── 03_deleted_lines/
    └── 04_deleted_blank/
```

## 三、清洗规则说明

### 3.1 噪声过滤（自动删除）

| 规则 | 示例 |
|---|---|
| 水印行（多种变体） | "仅供个人科研教学使用！" |
| 纯页码行 | "001" "002" |
| 全大写拼音行 | "ZHONGZI KE YIGE..." |
| 出版/版权关键词行 | "责任编辑：xxx" |
| ISBN / CIP 数据行 | "ISBN 978-7-..." |
| OCR 合并的出版团队行 | "出版人策划编辑项目统筹..." |
| 系列丛书行 | "教育家书院丛书·研究系列" |
| 孤立人名行（有内部空格的短行） | "刘 灿" / "何 薇 郑 莉" |
| 免责声明行 | "如有印装质量问题..." |
| 出版社信息表格（整表跳过） | 含"出版发行"/"社址"的表格 |
| 广告水印 | "更多教学好书请加微信：xxx" |
| 页眉重复行（章节标题单独成行） | 正文中重复出现的章标题 |

> **二次检测**：`fix_text()` 压缩空格后再过一遍噪声规则，
> 防止"出 版 人 策划编辑"这类被空格撑开的行漏网。
> 第一遍跑完后必须抽查，发现漏网噪声用正则二次清洗。

### 3.2 文本修复 + 格式标准化

| 问题 | 修复方式 |
|---|---|
| 章节末尾页码 "写在前面\|001" | 正则去掉 `\|\d+` 后缀 |
| OCR 汉字间多余空格 "种 子 课" | 检测中文占比>50% 且有单字间隔，则压缩空格 |
| 多余连续空格 | 压缩为单个空格 |
| 半角标点（中文语境） | `,` → `，`、`?` → `？`、`!` → `！`、`()` → `（）` |

### 3.3 结构识别（转为 Markdown 标题）

| 样式 | Markdown 级别 | 示例 |
|---|---|---|
| `一、二、三...` 开头的短行 | `##` | "二、种子课：用生长替代重复" |
| `*` 或 `·` 开头的导语行（含≥2中文字） | `###` | "* 导 语" |
| `（一）（二）` 括号编号行 | `####` | "（一）教学目标分析" |
| `——` 开头的副标题 | `>` 引用块 | "——以《用字母表示数》为例" |
| `—` 开头的引文署名（短行） | `>` 引用块 | "—叶圣陶" |

### 3.4 表格处理

- 出版社信息表格：**自动跳过**（第一格含"出版发行/社址"等关键词）
- 内容表格：转为标准 Markdown 表格格式
- 空白表格（有效单元格 < 3）：自动跳过

### 3.5 去重

- 段落级去重：`seen_lines` 集合检测重复段落（>10字符）
- 连续重复行检测：相邻两行相同则只保留一行
- 页眉重复行：章标题在正文中反复出现时，作为噪声删除

## 四、图片处理策略（v2 新增）

### 4.1 核心原则

- **不是全删也不是全留** — 按类型分类处理
- **文字描述 + 保留原图路径** — 文字保证 RAG 能搜到，原图留着以后用
- **忠实原文** — 看不清就写"扫描模糊"，绝不编造

### 4.2 自动分类规则

| 过滤条件 | 判定 | 处理 |
|---|---|---|
| 面积 < 5000px² | 装饰碎片（1px线、点） | 删除 |
| 长宽比 > 8:1 | 分隔线、边框线 | 删除 |
| 平均亮度 > 230 | 水印残片、空白扫描页 | 删除 |
| 平均亮度 < 10 且面积 < 50000px² | 纯黑装饰块 | 删除 |
| 以上都不满足 | 可能有内容 | 保留 |

### 4.3 人工审查（脚本跑完后必做）

自动分类会有漏网之鱼，跑完后必须：

1. 用 Read 逐张查看保留的图片
2. 分三类：
   - **垃圾**（书脊、空白页、灰色装饰块）→ 删除整行
   - **装饰可转文字**（签名、章节标题装饰）→ 替换为文字
   - **有教学价值**（插图、图表、公式图）→ 写描述 + 保留原图
3. 用 Bash 脚本批量替换 MD 中的图片引用

### 4.4 图片描述格式

```markdown
[图片描述：网格纸上的梯形，左半部分用斜线阴影填充，演示通过分割计算梯形面积的方法]
![图片](images/rId109.jpeg)
```

### 4.5 PDF 交叉参考

扫描模糊的图片，可以导出 PDF 对应页来辅助确认内容：

```python
import fitz  # pymupdf
doc = fitz.open("file.pdf")
page = doc[页码]
pix = page.get_pixmap(dpi=150)
pix.save("page_XXX.png")
```

**但必须忠实原文，不能产生幻觉。**

## 五、数学符号处理策略

| 符号类型 | 处理方式 | 原因 |
|---|---|---|
| `√ × ÷ ± → ℃ ≤ ≥ ≠` 等 Unicode | **直接保留** | Markdown/所有 RAG 系统都支持 |
| OMML 公式（Word 内嵌公式对象） | v2 可提取为图片 + 描述 | python-docx 不原生解析 |
| LaTeX 分数 `$\frac{1}{10}$` | 转中文语义"十分之一（1/10）" | 中文语义更利于 RAG 检索 |
| LaTeX 字符串 | 直接保留 `$...$` 格式 | 已是文本，无需转换 |

**如遇到 OMML 公式（`<m:oMath>` 标签）的处理思路：**

```python
# 检测是否含有 OMML
full_xml = doc.element.xml
has_omml = '<m:oMath' in full_xml
# 如果有，需要用 lxml + XSLT 转换为 LaTeX，或者安装 mammoth 库
```

## 六、RAG 注意事项

- **不去停用词** — 现代 Embedding 模型需要完整自然语言，去停用词反而降低检索质量
- **图片在标准 RAG 框架中不会被传给模型** — 所以必须在清洗阶段就写好文字描述
- **保留原图路径** — 未来 RAG 框架支持多模态时可直接用

## 九、常见问题

**Q：中文路径报错 PackageNotFoundError？**
A：不要用 bash heredoc 传路径，改用 `Write` 工具把 Python 脚本写成文件再运行。

**Q：某些人名/短行没被过滤掉（如"曲凤玲"）？**
A：二次清洗阶段用 `^[\u4e00-\u9fff]{2,3}$` 匹配孤立2-3字汉字行。但要注意保护标题行（以 # 开头的不删）。

**Q：表格内容有重复列？**
A：DOCX 合并单元格在 python-docx 中会重复，属正常现象，不影响语义。

**Q：图片过滤后还有垃圾图？**
A：自动分类不可能100%准确，必须人工逐张审查保留的图片，手动删除漏网垃圾。

**Q：扫描质量太差看不清图片内容？**
A：用 pymupdf 导出 PDF 对应页查看，PDF 通常比 Word 内嵌图片更清晰。但描述必须忠实，看不清就写"扫描模糊"。

## 十、处理复盘：《低头找幸福》经验总结（2026-03-17）

> 以下是处理俞正强《低头找幸福(OCR).docx》时发现的真实错误及改进策略。
> 每一条都是踩坑后写下的教训，以后处理新书前必须对照检查。

---

### 10.1 标题识别错误：只用文字正则，没有用字号

**错误**：脚本只用 `一、二、三...` 和 `（一）（二）` 等文字正则判断标题，导致全书 101 个标题里只识别出 10 个（漏掉了 `## 教育是一门艺术`、`#### 访谈手记` 等所有按字号设计的标题）。

**根本原因**：没有先做字号分析，直接套用了上一本书的文字正则，但本书标题完全靠字号区分。

**改进策略**：
1. 每处理一本新书，必须**先运行字号分析脚本**（`analyze_font_sizes.py`），获取各字号出现次数和示例文本
2. 根据分析结果设定 `SIZE_TO_HEADING` 字号映射，**字号识别优先于文字正则**
3. 参考文件 `四年级下册数据清洗提示词san三本.md` 中已有各书的字号配置

```python
SIZE_TO_HEADING = {
    23.5: 2,  # 教育是一门艺术
    16.0: 3,  # 1998—2002 忽如一夜春风来
    14.5: 4,  # ◎ 分数准备课教学设计
    13.5: 4,  # 访谈手记
}
```

---

### 10.2 OCR 字号漂移：同一标题被识别为多种字号

**错误**：`访谈手记` 在原书中理应是统一字号，但 OCR 将不同页面识别为 13.0、13.5、14.0、14.5pt 四种，`SIZE_TO_HEADING` 只映射了 13.5pt，**导致 32 处 `访谈手记` 没有 `####` 前缀**，在 Markdown 里成了裸文本行。

**根本原因**：OCR 扫描的字号识别本身不稳定，同一行字在不同页面的识别结果可能差 0.5~1pt。

**改进策略**：
1. **字号映射要放宽**：不要只映射单一精确值，相邻 ±0.5pt 的字号若语义相同，一起收进映射
2. **建立 `KNOWN_HEADINGS` 固定文本字典**：对于反复出现、语义固定的标题文本，无论字号多少都强制加标题前缀

```python
# 无论字号如何，这些文本固定为 #### 标题
KNOWN_HEADINGS_H4 = {'访谈手记', '苏霍姆林斯基'}

def detect_heading(text, font_size):
    t = text.strip()
    # 最高优先：固定文本（防OCR字号漂移）
    if t in KNOWN_HEADINGS_H4:
        return '#### ', t
    # 次优：字号映射
    if font_size and font_size in SIZE_TO_HEADING:
        ...
```

3. **清洗后必须统计各级标题数量**：如果 `##` 只有 4 个、`####` 只有 6 个，肯定有问题，正常书应该有几十到上百个标题

---

### 10.3 图片过滤阈值太激进，漏删了有用的图

**错误**：亮度阈值设为 230，但 rId90（亮度 230.9）被误删，而该图实际是有内容的表格页。

**根本原因**：自动分类阈值是统计经验值，边界附近的图片必然会有误判。

**改进策略**：
1. **不能只审查保留的图，必须也审查删除的图**，特别是边界附近（亮度 225~235、面积 3000~6000px²）的
2. 运行 `export_all_images.py` 生成图片审查报告（HTML），同时包含保留和删除的图片
3. 解压后双击 `审查报告.html` 用浏览器打开（**必须先解压再打开**，不能在 zip 里直接打开）
4. 人工核对删除的图中是否有误删，确认后告知 AI 哪些要加回来

---

### 10.4 图片描述错误：只描述外观，没有识别数学思维

**错误**：对 rId172（动物学校插图）的第一次描述写的是"椭圆形和三角形网络图"，没有识别出每只动物展示的是**哪种有序列举方法**（兔子=固定十位列举法，猴子=圆圈连线网络法，蛇=三角形连线法）。

**根本原因**：只从图片外观（颜色、形状）描述，没有站在**小学数学老师的视角**去追问"这张图在展示哪种数学思维方法"。

**改进策略**：
- 描述教学图时，必须先问：**"这在教什么？用什么方法？学生从中学到什么？"**，再写描述
- 描述结构：`①是什么场景 ②每个元素展示的数学方法是什么 ③这个图在书中的作用是什么`
- 如果看不清，**诚实写"扫描模糊"**，不要编造，让老师对照原书确认

---

### 10.5 噪声规则不全：中文关键词的空格变体漏网

**错误**：出版社信息表格里的"出版发行"在 OCR 中被识别为"出 版发行"（含空格），噪声规则用的是精确字符串匹配，漏掉了带空格的变体。

**改进策略**：
1. 出版信息表格过滤：用模糊匹配，把所有可能的空格变体都纳入
2. 页眉/页脚过滤：书名页眉可能有空格（如"俞正强 ： 低头找"），正则要加 `\s*` 处理
3. **目录行**：含 `/数字` 格式（如"第三章是金子总会发光的/46"），用正则 `.+/\d+\s*$` 过滤
4. **二次清洗必做**：第一轮清洗后，用专门的 `second_pass_*.py` 脚本再过一遍，针对本书特有的漏网噪声

---

### 10.6 OCR 孤立乱码：孤立字号段落是危险信号

**错误**：全书唯一一个 17pt 段落"对于可放一"，是 OCR 扫描产生的乱码，但没被任何噪声规则捕获，出现在正文里。

**改进策略**：
- **字号分析后，凡出现次数 ≤ 3 次、字号在全书孤立、文本语义不通的段落，必须人工确认**
- 判断方法：看字号分析表中次数极少（1~3次）的行，逐一确认其是否合理
- 如确定是乱码，加进 `second_pass_*.py` 的噪声规则，或直接从 MD 中删除

---

### 10.7 图片路径问题：相对路径无法供 AI 读取

**错误**：Markdown 中使用 `images/rId172.jpeg` 相对路径，在 Windows 下打开时 AI 无法通过路径访问本地图片文件。

**改进策略**：根据用途输出两个版本：

| 版本 | 路径格式 | 适用场景 |
|---|---|---|
| 本地查看版 | `images/rId172.jpeg`（相对路径） | Typora/Obsidian 打开，解压后和 images/ 放同一层 |
| AI 可读版 | GitHub raw URL | 粘贴给 AI 分析，或 RAG 系统读取 |

GitHub raw URL 格式：
```
https://github.com/仓库/raw/分支名/路径/images/rId172.jpeg
```
**前提：GitHub 仓库必须设为 public，否则 AI 无法访问**。

---

### 10.8 图片审查 HTML 报告设计教训

**错误**：
- 第一版：base64 内嵌图片，部分浏览器因安全策略不显示
- 第二版：HTML 用相对路径引用图片，用户在 zip 内直接打开 HTML，图片路径失效

**正确做法**：
1. HTML 用**相对路径**引用图片（不用 base64）
2. 在 HTML 标题区域加醒目提示：**"必须先解压 zip，再双击 HTML 文件打开"**
3. 解压后文件夹结构：
```
审查报告.html          ← 双击这个
01_kept/               ← HTML里引用 01_kept/rId172.jpeg
02_deleted_small/
...
```

### 10.9 清洗后必做的质量检查清单

每次清洗完成后，在提交给老师之前，必须完成以下检查：

- [ ] 标题数量是否合理（`## ≥10`、`#### ≥20`，视书的体量而定）
- [ ] 用短行扫描脚本检查是否有裸文本的标题/OCR乱码
- [ ] 字号分析表中是否有孤立字号（出现次数 ≤ 3）且文本语义不通的段落
- [ ] 图片审查：保留的图人工逐张确认，删除的图中边界附近的也要抽查
- [ ] 搜索 MD 中是否有明显语义断裂的短行（1~6字纯汉字行）
- [ ] 图片描述是否从**教学视角**而非外观视角写（特别是数学图）
- [ ] 是否有被误删的有价值图片需要加回来
